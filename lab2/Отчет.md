### Часть 5. Infrastructure Layer (Инфраструктурный слой)

#### Входящий адаптер: REST API

**RecipeController**:

```python
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel
from typing import List


class GenerateRecipeRequest(BaseModel):
    ingredient_names: List[str]


def setup_routes(app: FastAPI, generate_use_case, history_use_case):
    @app.post("/api/recipes/generate")
    async def generate_recipe(request: GenerateRecipeRequest, user_id: str = "user-123"):
        try:
            recipe = generate_use_case.generate(user_id, request.ingredient_names)
            return {"id": recipe.id, "name": recipe.name, "steps": recipe.steps}
        except Exception as e:
            raise HTTPException(status_code=400, detail=str(e))

    @app.get("/api/recipes/history")
    async def get_history(user_id: str = "user-123"):
        recipes = history_use_case.get_history(user_id)
        return [{"id": r.id, "name": r.name} for r in recipes]
```

**Эндпоинты**:
- `POST /api/recipes/generate` - Генерация рецепта из ингредиентов
- `GET /api/recipes/history` - Получение истории рецептов

#### Исходящий адаптер: Repository

**InMemoryRecipeRepository:

```python
from typing import Dict, List
from src.domain.models.recipe import Recipe


class InMemoryRecipeRepository:
    def __init__(self):
        self._storage: Dict[str, Recipe] = {}

    def save(self, recipe: Recipe) -> Recipe:
        self._storage[recipe.id] = recipe
        return recipe

    def find_by_user_id(self, user_id: str) -> List[Recipe]:
        return [r for r in self._storage.values() if r.user_id == user_id]
```

**Принцип работы**:
Данные хранятся в словаре _storage в оперативной памяти. При сохранении рецепт добавляется по ключу id. При поиске по user_id фильтруются все рецепты пользователя.

**Как работает DI (Dependency Injection):**

| `create_app()` | Фабрика приложения FastAPI, собирающая всё вместе |

### Что получилось хорошо

Удалось чётко отделить бизнес-логику (Recipe, Ingredient, User) от инфраструктуры. Domain слой не имеет зависимостей от FastAPI, базы данных или внешних сервисов. RecipeService легко тестируется с моками благодаря внедрению зависимостей через порты. Гексагональная архитектура позволила изолировать ядро системы и сделать его независимым от внешних технологий.

### С какими трудностями столкнулись

Было сложно понять, зачем создавать интерфейс для порта, если у него только одна реализация. После изучения принципа Dependency Inversion стало ясно: это для тестируемости и возможности замены адаптера (например, InMemory → PostgreSQL без изменения кода домена). Также потребовалось время на правильное разделение слоёв и понимание, какие операции должны быть синхронными, а какие асинхронными.

### Что узнали нового

Изучила принцип Dependency Inversion (DIP): модули верхнего уровня не зависят от модулей нижнего уровня, оба зависят от абстракций. Поняла, как гексагональная архитектура (Ports & Adapters) упрощает замену БД, AI-сервиса или способа уведомлений без изменения бизнес-логики. Освоила структуру проекта с разделением на domain, application и infrastructure. Научилась проектировать входящие и исходящие порты, реализовывать адаптеры и настраивать Dependency Injection.
