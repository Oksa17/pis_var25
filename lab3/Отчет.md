| 2 | `RecipeStatus` | Статус рецепта (DRAFT, GENERATED, COOKING, COMPLETED, FAILED, ARCHIVED) | Правила переходов между статусами | ✅ | `domain/value_objects/recipe_status.py` |

**Пример кода (Ingredient):**

```python
from dataclasses import dataclass
from datetime import date
from typing import Optional


@dataclass(frozen=True)
class Ingredient:
    """Value Object: Ингредиент (иммутабельный)"""
    name: str
    expiry_date: Optional[date] = None

    def __post_init__(self):
        if not self.name or len(self.name.strip()) == 0:
            raise ValueError("Название ингредиента не может быть пустым")
        if len(self.name) > 50:
            raise ValueError("Название ингредиента не может превышать 50 символов")

    def is_expired(self) -> bool:
        if self.expiry_date is None:
            return False
        return self.expiry_date < date.today()

    def is_available(self) -> bool:
        return not self.is_expired()
```

### 2. Entities (Сущности)

**Созданные Entity:**

1. **User** - пользователь системы, который генерирует рецепты
   - ID поле: `id` (UUID)
   - Бизнес-правила: лимит 10 запросов в час, отслеживание истории запросов
   - Файл: `domain/entities/user.py`

2. **Recipe** - рецепт блюда из доступных ингредиентов
   - ID поле: `id` (UUID)
   - Бизнес-правила: статусная модель с правилами переходов (DRAFT → GENERATED → COOKING → COMPLETED → ARCHIVED)
   - Файл: `domain/entities/recipe.py`

**Пример кода (User):**

```python
from dataclasses import dataclass, field
from datetime import datetime, timedelta
from typing import List


@dataclass
class User:
    """Entity: Пользователь (имеет ID)"""
    id: str
    email: str
    name: str
    role: str = "user"
    created_at: datetime = field(default_factory=datetime.now)
    _request_timestamps: List[datetime] = field(default_factory=list, repr=False)

    def __eq__(self, other):
        if not isinstance(other, User):
            return False
        return self.id == other.id

    def __hash__(self):
        return hash(self.id)

    def can_generate_recipe(self, limit: int = 10, window_minutes: int = 60) -> bool:
        """Проверка лимита запросов (инвариант: не более 10 в час)"""
        now = datetime.now()
        cutoff = now - timedelta(minutes=window_minutes)
        self._request_timestamps = [ts for ts in self._request_timestamps if ts > cutoff]
        return len(self._request_timestamps) < limit

    def add_generation_request(self) -> None:
        """Фиксация запроса на генерацию"""
        self._request_timestamps.append(datetime.now())

    def get_requests_count_last_hour(self) -> int:
        """Количество запросов за последний час"""
        now = datetime.now()
        cutoff = now - timedelta(minutes=60)
        return len([ts for ts in self._request_timestamps if ts > cutoff])
```

### 3. Aggregate Root (Корневой агрегат)

**Aggregate Root:** `CookingSession` (Сессия приготовления)

**Границы агрегата:**
- Корень: `CookingSession`
- Внутренние сущности: `User`, `Recipe`
- Value Objects: `CookingTime`, `CookingStep`, `Ingredient`, `RecipeStatus`

**Инварианты агрегата:**

| № | Инвариант | Как проверяется |
|---|----------|----------------|
| 1 | Нельзя генерировать рецепт при превышении лимита запросов (10 в час) | В методе `generate_recipe()` проверка `user.can_generate_recipe()` |
| 2 | Нельзя генерировать рецепт без ингредиентов | Проверка `if not ingredients` в `generate_recipe()` |
| 3 | Нельзя генерировать рецепт, если все ингредиенты просрочены | Проверка `if not fresh_ingredients` после фильтрации |
| 4 | Нельзя начать готовку, если рецепт не сгенерирован | Проверка `recipe.status` в `start_cooking()` |
| 5 | Шаги должны выполняться по порядку | Проверка `step_number == len(completed_steps) + 1` в `complete_step()` |
| 6 | Нельзя завершить несуществующий шаг | Проверка `step_number <= len(recipe.steps)` в `complete_step()` |

**Пример кода Aggregate Root:**

```python
from dataclasses import dataclass, field
from datetime import datetime
from typing import List, Optional
from uuid import uuid4

from ..entities.user import User
from ..entities.recipe import Recipe
from ..value_objects.cooking_time import CookingTime
from ..value_objects.ingredient import Ingredient
from ..events.recipe_generated_event import RecipeGeneratedEvent
from ..events.cooking_started_event import CookingStartedEvent
from ..events.step_completed_event import StepCompletedEvent


@dataclass
class CookingSession:
    """Aggregate Root: Сессия приготовления"""
    id: str = field(default_factory=lambda: str(uuid4()))
    user: Optional[User] = None
    recipe: Optional[Recipe] = None
    cooking_time: Optional[CookingTime] = None
    completed_steps: List[int] = field(default_factory=list)
    _events: List = field(default_factory=list, repr=False)

    def _add_event(self, event) -> None:
        """Добавить доменное событие"""
        self._events.append(event)

    def get_events(self) -> List:
        """Получить и очистить события"""
        events = self._events.copy()
        self._events.clear()
        return events

    def generate_recipe(self, user: User, ingredients: List[Ingredient]) -> Recipe:
        # Инвариант 1: лимит запросов
        if not user.can_generate_recipe():
            raise ValueError(f"Превышен лимит запросов. Можно не более 10 в час")
        
        # Инвариант 2: не пустые ингредиенты
        if not ingredients:
            raise ValueError("Нельзя генерировать рецепт без ингредиентов")
        
        # Инвариант 3: не все просрочены
        fresh_ingredients = [i for i in ingredients if i.is_available()]
        if not fresh_ingredients:
            raise ValueError("Все ингредиенты просрочены!")
        
        # Создаём рецепт
        recipe = Recipe(
            user_id=user.id,
            name=f"Блюдо из {fresh_ingredients[0].name}",
            ingredients=fresh_ingredients,
            steps=[]
        )
        
        user.add_generation_request()
        self.user = user
        self.recipe = recipe
        
        # Доменное событие
        self._add_event(RecipeGeneratedEvent(
            recipe_id=recipe.id,
            user_id=user.id,
            ingredient_names=[i.name for i in fresh_ingredients]
        ))
        
        return recipe

    def start_cooking(self, recipe: Recipe, estimated_minutes: int) -> None:
        # Инвариант 4: рецепт должен быть сгенерирован
        if recipe.status.value != "сгенерирован":
            raise ValueError(f"Нельзя начать готовку рецепта в статусе {recipe.status.value}")
        
        self.recipe = recipe
        self.cooking_time = CookingTime(
            start_time=datetime.now(),
            estimated_duration_minutes=estimated_minutes
        )
        recipe.start_cooking()
        
        # Доменное событие
        self._add_event(CookingStartedEvent(
            recipe_id=recipe.id,
            user_id=recipe.user_id,
            estimated_minutes=estimated_minutes
        ))

    def complete_step(self, step_number: int) -> None:
        if not self.recipe:
            raise ValueError("Нет активной сессии готовки")
        
        # Инвариант 5: шаги по порядку
        if step_number != len(self.completed_steps) + 1:
            raise ValueError(f"Шаги должны выполняться по порядку. Следующий шаг: {len(self.completed_steps) + 1}")
        
        # Инвариант 6: шаг существует
        if step_number > len(self.recipe.steps):
            raise ValueError(f"Шаг {step_number} не существует. Всего шагов: {len(self.recipe.steps)}")
        
        self.completed_steps.append(step_number)
        
        # Доменное событие
        self._add_event(StepCompletedEvent(
            recipe_id=self.recipe.id,
            step_number=step_number,
            step_description=self.recipe.steps[step_number - 1].description if self.recipe.steps else "",
            completed_at=datetime.now()
        ))
        
        # Проверка завершения всех шагов
        if len(self.completed_steps) == len(self.recipe.steps):
            self.recipe.complete_cooking()
```

### 4. Domain Events (Доменные события)

**Созданные события:**

1. **RecipeGeneratedEvent** - генерируется после успешной генерации рецепта
   - Данные: `recipe_id`, `user_id`, `ingredient_names`, `occurred_at`
   - Файл: `domain/events/recipe_generated_event.py`

   - Данные: `recipe_id`, `user_id`, `estimated_minutes`, `started_at`
   - Файл: `domain/events/cooking_started_event.py`

3. **StepCompletedEvent** - генерируется при завершении каждого шага готовки

4. **IngredientMissingEvent** - генерируется при обнаружении отсутствующего ингредиента
   - Данные: `recipe_id`, `missing_ingredient`, `available_ingredients`, `occurred_at`
   - Файл: `domain/events/ingredient_missing_event.py`

## Контрольные вопросы

**1. В чём отличие Value Object от Entity?**

Value Object не имеет собственной идентичности и определяется исключительно своими атрибутами. Два Value Object с одинаковыми значениями полей считаются равными. Entity имеет уникальный идентификатор (id) и сохраняет свою идентичность даже при изменении атрибутов.

| Характеристика | Value Object | Entity |
|----------------|--------------|--------|
| Идентичность | Нет (по значениям) | Есть (по ID) |
| Иммутабельность | Обычно да | Нет |
| Пример в системе | Ingredient (соль == соль) | User (id=123) |

**2. Почему Aggregate Root должен инкапсулировать доступ к внутренним сущностям?**

Aggregate Root гарантирует соблюдение инвариантов всего агрегата. Если внешний код получит прямой доступ к внутренним сущностям, он может изменить их состояние в обход бизнес-правил, что нарушит целостность данных.

**Пример из моей системы:** `CookingSession` инкапсулирует доступ к `Recipe`. Внешний код не может напрямую вызвать `recipe.start_cooking()` — только через `cooking_session.start_cooking()`, где проверяется, что рецепт находится в правильном статусе.

**3. Какая роль Domain Events? Приведите пример из вашей системы.**

Domain Events уведомляют другие части системы о важных изменениях в домене, не создавая жёстких связей. Они позволяют реализовать слабую связанность и eventually consistency.

**Пример из моей системы:**
- `RecipeGeneratedEvent` — после генерации рецепта подписчик может отправить уведомление пользователю
- `StepCompletedEvent` — после завершения шага подписчик может запустить таймер следующего шага
- `IngredientMissingEvent` — при отсутствии ингредиента подписчик может предложить альтернативу

**4. Как вы проверяете инварианты в вашем агрегате? Приведите пример.**

Инварианты проверяются в публичных методах агрегата перед изменением состояния. Если инвариант нарушен, метод выбрасывает исключение.

**Пример из моей системы:**

```python
def complete_step(self, step_number: int) -> None:
    # Инвариант: шаги должны выполняться по порядку
    if step_number != len(self.completed_steps) + 1:
        raise ValueError(f"Шаги должны выполняться по порядку. "
                        f"Следующий шаг: {len(self.completed_steps) + 1}")
    
    # Инвариант: шаг должен существовать
    if step_number > len(self.recipe.steps):
        raise ValueError(f"Шаг {step_number} не существует. "
                        f"Всего шагов: {len(self.recipe.steps)}")
    
    # Инвариант: нельзя завершить уже завершённый шаг
    if step_number in self.completed_steps:
        raise ValueError(f"Шаг {step_number} уже завершён")
    
    # Если все проверки пройдены — выполняем действие
    self.completed_steps.append(step_number)
```

**5. Почему Value Objects делаются иммутабельными?**

Иммутабельность Value Objects гарантирует, что объект нельзя изменить после создания. Это важно, потому что:

1. **Безопасность** — два объекта с одинаковыми значениями должны быть взаимозаменяемы
2. **Предсказуемость** — Value Object не меняется неожиданно
3. **Побочные эффекты** — иммутабельные объекты можно безопасно передавать между компонентами
4. **Хэширование** — иммутабельные объекты могут использоваться как ключи в словарях

**Пример из моей системы:**

```python
@dataclass(frozen=True)
class Ingredient:
    """Value Object: Ингредиент (иммутабельный)"""
    name: str
    expiry_date: Optional[date] = None

    def __post_init__(self):
        if not self.name or len(self.name.strip()) == 0:
            raise ValueError("Название ингредиента не может быть пустым")

    def is_expired(self) -> bool:
        if self.expiry_date is None:
            return False
        return self.expiry_date < date.today()


# Попытка изменить Value Object вызовет ошибку
ingredient = Ingredient(name="соль")
# ingredient.name = "перец"  # FrozenInstanceError: cannot assign to field 'name'
```

### С какими трудностями столкнулись

Было сложно правильно спроектировать границы агрегата `CookingSession` — какие сущности должны быть внутри (User, Recipe), а какие снаружи. Также потребовалось время на реализацию правил переходов между статусами рецепта (DRAFT → GENERATED → COOKING → COMPLETED → ARCHIVED) и проверку порядка выполнения шагов готовки (шаг 1 → шаг 2 → шаг 3).

### Что узнали нового

Изучила тактические паттерны DDD:
- **Value Objects** — иммутабельность (frozen=True), валидация в `__post_init__`, сравнение по значениям
- **Entity** — identity-based equality (сравнение по `id`), переопределение `__eq__` и `__hash__`
- **Aggregate Root** — границы агрегата, инкапсуляция внутренних сущностей, проверка инвариантов
- **Domain Events** — регистрация событий при изменении состояния, слабая связанность компонентов

Поняла, как доменный слой отличается от CRUD-подхода — бизнес-правила находятся внутри домена (в методах агрегатов и Value Objects), а не в сервисах или контроллерах.
