### 1. Repository (PostgreSQL)

**Реализованные методы:**

| № | Метод | Описание |
|---|-------|----------|
| 1 | `save(recipe)` | Сохранение или обновление рецепта в БД |

### 2. REST Controller

**Эндпоинты:**

| № | Метод | Path | Описание |
|---|-------|------|----------|
| 1 | POST | `/api/recipes/generate` | Сгенерировать рецепт из ингредиентов |
| 2 | POST | `/api/recipes/{id}/start` | Начать приготовление рецепта |
| 3 | POST | `/api/sessions/{id}/step/{step}` | Завершить шаг приготовления |
| 4 | POST | `/api/recipes/{id}/rate` | Оценить рецепт (1-5★) |
| 5 | GET | `/api/recipes/{id}` | Получить рецепт по ID |
| 6 | GET | `/api/users/{id}/history` | История рецептов пользователя |
| 7 | DELETE | `/api/recipes/{id}` | Удалить рецепт |

**Скриншот Postman/Swagger:**

### 3. Docker Compose

**Сервисы:**

| № | Сервис | Назначение |
|---|--------|------------|
| 1 | `db` | PostgreSQL 15 (основная БД) |
| 2 | `app` | FastAPI приложение (рецепто-генератор) |

**docker-compose.yml:**

```yaml
version: '3.8'

services:
  db:
    image: postgres:15
    container_name: recipe_db
    environment:
      POSTGRES_USER: recipe_user
      POSTGRES_PASSWORD: recipe_password
      POSTGRES_DB: recipe_db
    ports:
      - "5432:5432"
    volumes:
      - postgres_data:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U recipe_user -d recipe_db"]
      interval: 5s
      timeout: 5s
      retries: 5

  app:
    build: .
    container_name: recipe_app
    ports:
      - "8000:8000"
    environment:
      DATABASE_URL: postgresql://recipe_user:recipe_password@db:5432/recipe_db
    depends_on:
      db:
        condition: service_healthy
    volumes:
      - ./app:/app

volumes:
  postgres_data:
```

## Контрольные вопросы

**1. Почему Repository находится в Infrastructure, а не в Domain?**

Repository находится в Infrastructure, потому что он отвечает за технические детали хранения данных (подключение к БД, SQL-запросы, транзакции), а не за бизнес-логику. Domain слой определяет только интерфейс (порт) репозитория, а реальная реализация подключается через Dependency Injection. Это遵循 принципу Dependency Inversion (DIP) — домен не зависит от инфраструктуры.

**Пример из моей системы:**

| Слой | Компонент | Ответственность |
|------|-----------|-----------------|
| Domain | `RecipeRepository` (интерфейс) | Определяет методы `save()`, `find_by_id()` |
| Infrastructure | `PostgresRecipeRepository` (реализация) | Работа с SQLAlchemy и PostgreSQL |

**2. В чём преимущество ORM над обычным SQL?**

| № | Преимущество | Описание |
|---|--------------|----------|
| 1 | **Абстракция от БД** | Работа с объектами Python, а не с таблицами и SQL-запросами |
| 2 | **Безопасность** | Автоматическая защита от SQL-инъекций через параметризованные запросы |
| 3 | **Переносимость** | Легко сменить PostgreSQL на MySQL или SQLite без изменения кода |
| 4 | **Управление схемой** | Автоматическая генерация миграций (Alembic) при изменении моделей |
| 5 | **Производительность** | Ленивая загрузка (lazy loading), кэширование, решение проблемы N+1 |
| 6. | **Поддержка транзакций** | Упрощённая работа с транзакциями через контекстные менеджеры |
