# Анализ границ ответственности для сценария «Создание рецепта из того, что есть»

## Транзакционные границы

### 1. Где начинается и заканчивается транзакция?
Транзакция начинается в момент получения запроса `POST /api/recipes/generate` контроллером `RecipeService` и заканчивается после фиксации записи о сгенерированном рецепте в базе данных (история запросов пользователя). Успешная фиксация рецепта в истории является точкой невозврата для основного доменного события «Рецепт предложен».

### 2. Какие операции должны быть атомарными?
- Проверка списка ингредиентов на просрочку.
- Запрос к AI-сервису на генерацию рецепта.
- Сохранение рецепта в историю пользователя.
- Сохранение чекаута готовки (чек-листа) в кэш.

Все эти операции должны быть выполнены в рамках одной логической транзакции, чтобы исключить ситуацию, когда AI сгенерировал рецепт, но пользователь его не получил (ошибка сохранения).

### 3. Какие операции могут быть асинхронными?
- **Отправка уведомления «Время перевернуть блюдо».** Это не влияет на факт генерации рецепта. Если уведомление зависнет, готовка не сорвётся.
- **Сохранение аналитики (пользователь оценил рецепт, лайк/дизлайк).** Это не критично для доменной операции.
- **Проверка лимита запросов (10 в час).** Может выполняться с небольшой задержкой через eventual consistency.

### 4. Анализ операций

| Операция | Синхронная/Асинхронная | Откат при ошибке | Retry-стратегия | Идемпотентность |
| :--- | :--- | :--- | :--- | :--- |
| **Валидация входных данных (ингредиенты не пустые, не более 20 позиций)** | Синхронная | Да (остановка потока, HTTP 400) | Нет | Нет (зависит от входных данных) |
| **Проверка просрочки ингредиентов по БД сроков годности** | Синхронная | Да (ROLLBACK, если не прошли проверку) | Нет (ошибка данных фатальна) | Да (одна и та же проверка для одного списка ингредиентов — одинаковый результат) |
| **Запрос к AI-сервису на генерацию рецепта** | Синхронная (для пользователя) | Да (не фиксируем рецепт, если AI упал) | 2 попытки с задержкой 1s, затем fallback на кэш топ-5 рецептов | Да (одинаковый список ингредиентов → одинаковый хэш, кэшировать на 1 час) |
| **Сохранение рецепта в историю (PostgreSQL)** | Синхронная | Да (ROLLBACK всей транзакции) | Нет (ошибка БД фатальна, отдаём 500) | Да (уникальный `request_id` на клиенте) |
| **Проверка лимита запросов (10 в час)** | Синхронная | Да (отказ при превышении, код 429) | Нет | Да (считаем по `user_id` за временное окно) |
| **Отправка push-уведомления «Пора перевернуть гренки»** | Асинхронная (через WebSocket/SSE) | Нет (best-effort, пользователь сам вспомнит) | 3 попытки через 2s, 4s, 8s, затем в DLQ | Да (дедупликация по `recipe_id` + `step_number` в очереди) |
| **Сохранение оценки пользователя (лайк/дизлайк)** | Асинхронная | Нет (аналитика не критична для домена) | 5 попыток, затем сброс с логированием | Да (одна оценка на `user_id` + `recipe_id`) |

## Обработка исключительных ситуаций

### Исключительная ситуация 1: Таймаут при запросе к AI-сервису

**Условие**: Внешний AI-сервис (например, OpenAI или локальный GPT) не отвечает в течение 3 секунд.

**Обнаружение**:
- HTTP-клиент выбрасывает `TimeoutException` или `ReadTimeoutException`.
- Логируется ошибка: "AI service timeout for ingredients: {ingredient_list}".

**Реакция**:
1. Система НЕ сохраняет рецепт в историю.
2. Система прерывает текущую транзакцию.
3. Система пытается 2 раза повторить запрос с экспоненциальной задержкой (1s, 2s).
4. При повторном таймауте — переключается на **кэш топ-5 базовых рецептов** (жареный лук, бутерброд, гренки, омлет, салат из помидоров).
5. Система помечает ответ как `"source": "fallback_cache"`.

**Компенсация**:
- Если AI упал полностью, запускается фоновый шедулер, который раз в 5 минут проверяет доступность AI и восстанавливает кэш.
- Пользователю показывается уведомление: "AI сейчас занят, держи классический рецепт".

**Уведомление пользователя**:
- Пользователь видит рецепт из кэша с пометкой «Базовый режим».

---

### Исключительная ситуация 2: Ошибка подключения к очереди уведомлений (RabbitMQ/Kafka)

**Условие**: После сохранения рецепта в историю, система не может опубликовать задачу на отправку push-уведомления «Пора перевернуть блюдо».

**Обнаружение**:
- При вызове `rabbitTemplate.convertAndSend()` выбрасывается `AmqpException` или `KafkaException`.

**Реакция**:
1. Система логирует критическую ошибку.
2. Система НЕ откатывает транзакцию БД (рецепт уже сгенерирован и показан пользователю).
3. Система помещает задачу на отправку уведомления в **локальную таблицу `outbox`** со статусом `PENDING`.

**Компенсация**:
- Запускается фоновый процесс (Scheduler), который раз в 30 секунд сканирует таблицу `outbox` и пытается повторно отправить неотправленные сообщения в очередь.
- После успешной отправки статус в `outbox` меняется на `SENT`.
- Если через 10 минут отправка не удалась — уведомление помечается как `FAILED` и отправляется алерт админу.

**Уведомление пользователя**:
- Пользователь видит сообщение об успешной генерации рецепта, так как доменная операция выполнена.
- Уведомление «Пора перевернуть» может не прийти (но это не фатально).
- Системный администратор получает алерт: «Очередь уведомлений недоступна».

### Исключительная ситуация 3: Превышен лимит запросов (10 рецептов за час)

**Условие**: Пользователь отправил 11-й запрос на генерацию рецепта в течение часа.

**Обнаружение**:
- Система проверяет счётчик запросов по `user_id` в Redis или БД.
- Счётчик показывает 10.

**Реакция**:
1. Система НЕ начинает транзакцию генерации рецепта.
2. Система возвращает HTTP 429 (Too Many Requests).
3. Система добавляет заголовок `Retry-After: 3600` (или время до конца часа).

**Компенсация**:
- Счётчик автоматически обнуляется через час (TTL в Redis).
- Пользователю предлагается подождать или купить расширенный тариф (если есть монетизация).

**Уведомление пользователя**:
- Пользователь видит сообщение: «Ты уже сгенерировал 10 рецептов за час. Отдохни, кухня остывает. Вернись через {оставшиеся_минуты} минут».

---

### Исключительная ситуация 4: Просроченный ингредиент в холодильнике

**Условие**: В списке ингредиентов пользователя есть продукт с датой годности < текущей даты.

**Обнаружение**:
- Сервис проверки свежести (`FreshnessService`) запрашивает БД сроков годности.
- Найден ингредиент со статусом `EXPIRED`.

**Реакция**:
1. Система НЕ отправляет запрос в AI.
2. Система удаляет просроченный ингредиент из списка.
3. Система добавляет предупреждение в ответе: `"warnings": ["Яйца просрочены на 3 дня, не рискуй!"]`.
4. Система генерирует рецепт на оставшихся ингредиентах.

**Компенсация**:
- Если после удаления просрочки осталось 0 ингредиентов → рецептов нет → отдаём совет «Сходи в магазин».
- Пользователь может подтвердить игнорирование предупреждения (но система всё равно не рекомендует).

**Уведомление пользователя**:
- Пользователь видит красный текст: «Внимание! Яйца просрочены. Я их не использую. Выброси их, пожалуйста 🗑️».

## Почему «на потом» — плохо (вывод)

| Что откладываем | К чему приводит |
|----------------|----------------|
| Проверку просрочки «на потом» | Отравление пользователя 🦠 |
| Лимит запросов «на потом» | DDoS своими же пользователями, убитый AI-сервис |
| Retry для AI-таймаута «на потом» | Пользователь уходит без рецепта и больше не возвращается |
| Падение очереди уведомлений «на потом» | Сгоревшая еда, потому что не напомнили перевернуть |

> **Железное правило:** Границы проверяются до начала транзакции. Откаты проектируются сразу. Асинхронность — только для недоменных операций.
